Introduction to IoT

Definition · Characteristics · Physical & Logical Design · Protocols · IoT Levels
Definition
IoT = A dynamic global network infrastructure with self-configuring capabilities based on standard and interoperable communication protocols, where physical and virtual "things" have identities, physical attributes, and virtual personalities and use intelligent interfaces.
Characteristics of IoT

Dynamic & Self-Adapting

Devices adapt to changing contexts.

Self-Configuring

Devices configure themselves automatically.

Interoperable Protocols

Communicate across different standards.

Unique Identity

Each device has a unique ID (IP/URI).

Integrated into Information Network

Seamlessly part of the internet.

  • I/O interfaces for sensors & actuators

Communication Models

Request-Response

Client sends request → Server responds. (HTTP/REST). Stateless.

Publish-Subscribe

Publishers → Broker manages topics → Consumers subscribe. (MQTT)

Push-Pull

Producers push data to queues → Consumers pull. Queues act as buffers.

Exclusive Pair

Bidirectional, full-duplex, persistent connection. Client closes. (WebSocket)
REST vs WebSocket APIs

REST API

HTTP Commands: GET, PUT, POST, DELETE. Payload: JSON/XML. Follows Request-Response model.

WebSocket API

Bidirectional full-duplex. Initial HTTP handshake → persistent WebSocket connection. Follows Exclusive Pair model.

IoT Architecture & M2M Communication

IoT Architecture layers · M2M · SDN · Cloud · Fog
IoT Architecture (3-Layer)

Perception Layer

Physical layer – sensors, RFID, actuators. Collects data from environment.

Network Layer

Transmits data via internet, cloud, Wi-Fi, Bluetooth, 3G/4G.

Application Layer

Delivers application-specific services to user (Smart Home, Healthcare, etc.).
5-Layer Architecture
LayerFunction
Business LayerManages business models, graphs, and user privacy
Application LayerSmart services (smart home, smart health)
Processing (Middleware)Stores, analyzes, and processes data; database & cloud computing
Transport LayerTransfers data between perception and processing layers (WiFi, 3G, LAN)
Perception LayerSensors and actuators collecting physical data
M2M (Machine to Machine)
  • Direct communication between machines without human intervention
  • Uses wired (Ethernet) or wireless (Wi-Fi, cellular) networks
  • Key components: sensors, RFID, Wi-Fi/cellular connection, autonomic computing
  • Applications: Smart metering, fleet management, remote monitoring
  • Difference from IoT: M2M = point-to-point; IoT = broader ecosystem with cloud & analytics
Cloud Computing in IoT

IaaS

Infrastructure as a Service – virtual machines, storage

PaaS

Platform as a Service – development platforms

SaaS

Software as a Service – apps over internet
Fog Computing (Edge Computing)
  • Processing done closer to IoT devices (at the "edge") rather than in the cloud
  • Reduces latency, bandwidth usage, and dependency on cloud
  • Useful for real-time applications
  • Fog = between device and cloud
SDN (Software Defined Networking) in IoT
  • Separates control plane from data plane
  • Central controller manages network behavior
  • Makes IoT networks more flexible, programmable, and manageable

Networks & Communication Aspects

Wireless Issues · MAC · WSN · Unicast/Multicast · RTS/CTS
Wireless Medium Access Issues

1. Half Duplex Operation

One party communicates at a time. During transmission, signal leakage prevents simultaneous reception. Collision detection by sender is impossible.

2. Time-Varying Channel

Three mechanisms:
• Reflection – wave hits large object
• Diffraction – wave bends around sharp edges
• Scattering – wave passes through objects smaller than wavelength

3. Burst Channel Errors

Contiguous sequence of erroneous symbols. First & last symbol have errors. Bit error rate in wireline = 10⁻³.
MAC Protocol Types

Contention-Based (Random Access)

CSMA/CD, CSMA/CA, ALOHA. Nodes compete for channel.

Contention-Free (Scheduled)

TDMA, FDMA, CDMA. Time/frequency/code slots assigned.

Ad Hoc MAC

Nodes communicate peer-to-peer. Each node also acts as router. More robust against failure.

Channelization Protocol

Bandwidth shared in time (TDMA), frequency (FDMA), or code (CDMA).
WSN (Wireless Sensor Network)
  • Infrastructure-less wireless network deployed with sensors in ad-hoc manner
  • Sensor nodes have onboard processors + connect to Base Station
  • Deployed in remote/hazardous locations for unattended operations

Components of WSN

  • Sensors – capture environmental data
  • Radio Nodes – microcontroller + transceiver + memory
  • WLAN Access Point – receives wireless data
  • Evaluation Software – processes & presents data

WSN Characteristics

  • Resource constrained (battery-powered)
  • Data-centric communication
  • Application specific design
  • Node failure & unreliable communication
  • Scalable & densely deployable
  • Dynamic topologies
WSN Applications & Challenges

Applications

IoT, Surveillance, Environmental monitoring, Noise level, Medical/patient monitoring, Agriculture, Landslide Detection

Challenges

QoS, Security, Energy Efficiency, Network Throughput, Performance, Node failure handling, Cross-layer optimization, Scalability
Unicast vs Multicast

Unicast

One-to-one. Single sender → single recipient. Most common form of data transfer. E.g. 10.1.4.0 → 20.14.4.2

Multicast

One-to-many or many-to-many. Sender → multiple targeted receivers simultaneously. Reduces bandwidth vs sending individual copies. Different from broadcast (broadcast = all nodes).
RTS / CTS (Flow Control)

RTS/CTS – Request to Send / Clear to Send

DTE (host) sends RTS → "I want to send data"
DCE (modem) replies CTS → "Go ahead, send"
Prevents collision in shared wireless channel.

DTR/DSR – Data Terminal Ready / Data Set Ready

DTE sends DTR → "I am ready to communicate"
DCE sends DSR → "I am ready"
Used to establish communication channel.
Data Aggregation & Dissemination
  • Data Aggregation – Combining data from multiple sensors to reduce redundancy and save bandwidth
  • Data Dissemination – Spreading information/commands from base station to sensor nodes
  • Key for energy efficiency in WSN

Web Infrastructure for Managing IoT Resources

IoT Management · Lifecycle · Scheduling · Device/Cloud Collaboration
IoT System Management
  • Manages connected devices, their data, and services across the IoT ecosystem
  • Uses OpenIoT architecture for IoT/Cloud convergence
  • Handles provisioning, monitoring, configuration, updates, and decommissioning
IoT Product Lifecycle (4 Stages)

1. Design Stage (Planning)

First & most critical stage. Plan architecture, security, performance. Decisions: make vs buy, integration, scalability.

2. Deployment Stage (Implementation)

PoC → Pilot → Full rollout. Multiple stakeholders. Fast provisioning, access control, large-scale setup.

3. Ongoing Management (Operation)

Longest stage. Remote monitoring, maintenance, updates, optimization. Data-driven decisions without physical visits.

4. Decommissioning (End-of-Life)

Often ignored. Securely remove old devices and onboard new ones. Plan from design stage for smooth transitions.
Example – Smart Electricity Meter: Design → Choose meter/network → Deploy in homes → Monitor billing remotely → Replace old meters
Scheduling in IoT
  • Goal: Maximize CPU utilization & throughput; Minimize latency & power consumption
  • Objectives:
    • Conflict-free time slot assignment
    • Priority based on data rate and delay requirements
    • Duty cycle adjusted based on priority, buffer size, energy, and transmit power
Resource Management in IoT
  • Physical resources = IoT nodes identified by URI
  • Resources include: computational, storage, energy, network/communication
  • Efficient management = discovering, identifying, selecting, partitioning, provisioning resources
  • Challenges:
    • Heterogeneity of IoT resources
    • Dynamic nature of IoT ecosystems
    • Resource-constrained IoT nodes
    • Failures causing performance degradation
  • Steps: Resource/Service Discovery → Resource Partitioning → Higher Utilization
Device/Cloud Collaboration
  • IoT devices collect data → send to cloud → processed & sent back
  • Applications: Smart watches, smart home, smart cities, self-driving cars, thermostats
  • Cloud examples: Antivirus apps, data processing, email, streaming, web conferencing

Challenges in IoT

Security · Design · Deployment · Common Challenges & Solutions
A. Security Challenges

1. Lack of Encryption

IoT devices lack storage/processing of traditional computers. Hackers manipulate protection algorithms. Data transmitted unencrypted.

2. Insufficient Testing & Updating

23B+ devices globally. Manufacturers rush to market. Devices become insecure over time. Solution: Proper testing before launch + regular updates.

3. Brute-Forcing & Default Passwords

Mirai Botnet – used default passwords for massive DDoS attacks. Consumers never change default credentials.

4. IoT Malware & Ransomware

Ransomware locks users out of devices using encryption. Hackers hijack cameras, demand ransom. Increases with device count.

5. IoT Botnet → Cryptocurrency

Botnet workers manipulate data privacy in open crypto market. Blockchain itself not vulnerable but app development is.
B. Design Challenges

1. Battery Life

Small chips require low weight & power. Screen sizes growing to fit larger batteries. Battery energy stays same.

2. Cost & Time to Market

Embedded systems cost-constrained. Need optimal design approaches. Bring device to market on time.

3. System Security

Must be robust & reliable. Requires cryptographic algorithms. Secure all components from prototype to deployment.
C. Deployment Challenges

1. Connectivity

Foremost concern. Poor connectivity = challenge for IoT sensors to monitor and supply info.

2. Cross-Platform Capability

Must balance hardware & software. Ensure best performance across all platforms despite heavy device loads.

3. Data Collection & Processing

Data plays important role. Need proper planning for collection, storage, and processing.

4. Lack of Skill Set

Skilled resources needed for IoT development. Right talent = key asset.

Developing IoT & IoT Tools

Raspberry Pi · Arduino · Embedded C · Python Programs
⭐ Raspberry Pi (PYQ)

Key Features

  • Credit-card sized single-board computer
  • ARM Processor – efficient, runs various OS
  • 40 GPIO Pins – connect sensors, LEDs, motors
  • HDMI + USB ports
  • Linux Operating System (Raspberry Pi OS)
  • SD card slot, RJ45 Ethernet, 3.5mm AV jack
  • RAM, ROM, Microprocessor on board

Varieties

  • Raspberry Pi 4 – most powerful, suitable for desktop
  • Raspberry Pi Pico – microcontroller, embedded/smaller projects

Why Use It?

  • Affordable vs traditional computers
  • Versatile – many applications
  • Learning platform for programming & electronics
  • Large maker community with tutorials
⭐ Arduino Board – Working Principles (PYQ)

Hardware Components

  • Microcontroller – tiny computer on chip; reads input, processes, controls output
  • Analog Pins (A0–A5) – reads analog voltage signals
  • Digital Pins (0–13) – reads/writes digital HIGH/LOW signals
  • Power Supply – USB cable or battery pack
  • Voltage regulators, USB connectors, communication interfaces

Software (Arduino IDE)

  • Free, user-friendly IDE
  • Write code in simplified C++
  • setup() – runs once, initializes pins
  • loop() – runs repeatedly forever
  • pinMode(pin, OUTPUT/INPUT)
  • digitalWrite(pin, HIGH/LOW)
  • delay(ms) – pause execution

Advantages

Simple & Accessible · Open-source · Scalable & Flexible

Workflow: 5 Steps

1. Project Idea → 2. Hardware Selection → 3. Code in Arduino IDE → 4. Upload via USB → 5. Power up & run
⭐ Arduino Program: LED Blinking + Buzzer (PYQ)
// Define pins const int ledPin = 13; const int buzzerPin = 8; // active-LOW buzzer void setup() { pinMode(ledPin, OUTPUT); pinMode(buzzerPin, OUTPUT); } void loop() { // LED ON + Buzzer ON digitalWrite(ledPin, HIGH); digitalWrite(buzzerPin, LOW); // active-LOW = LOW turns ON delay(1000); // LED OFF + Buzzer OFF digitalWrite(ledPin, LOW); digitalWrite(buzzerPin, HIGH); delay(1000); }
Key: HIGH = 5V = LED ON. Active-LOW buzzer: LOW = 0V = buzzer ON. delay(1000) = 1 second pause. loop() repeats forever creating blink+beep effect.
⭐ Embedded C: Room Temperature Sensor (PYQ)
#include <DHT.h> #define DHTPIN 3 // Digital pin connected to DHT sensor #define DHTTYPE DHT22 // DHT 22 (AM2302) DHT dht(DHTPIN, DHTTYPE); void setup() { Serial.begin(9600); dht.begin(); } void loop() { delay(2000); // Wait 2 seconds between readings float temperature = dht.readTemperature(); if (isnan(temperature)) { Serial.println("Failed to read from DHT sensor!"); return; } Serial.print("Temperature: "); Serial.print(temperature); Serial.println("°C"); }
DHT = Digital Humidity & Temperature sensor. readTemperature() reads Celsius. isnan() checks if reading is invalid (Not a Number). Serial Monitor displays output at 9600 baud rate.
⭐ Python: Search Dictionary by Key (PYQ)
# Define a dictionary my_dict = {'name': 'John', 'age': 30, 'city': 'New York'} # Key to search key_to_search = 'age' # Check if key exists if key_to_search in my_dict: print(f"Value for '{key_to_search}': {my_dict[key_to_search]}") else: print(f"Key '{key_to_search}' does not exist.")
Uses in operator to check key existence. If found, access with my_dict[key]. Output: Value for 'age': 30
